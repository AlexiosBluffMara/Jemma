## Training Datasets

### construction-cracks-train — 75 records
Expert-written Q&A pairs covering concrete crack classification and structural assessment.

**Topics:** Flexural, shear, torsion, ASR, D-cracking, settlement, thermal, plastic/drying shrinkage, crazing, corrosion-induced cracking. Bridge deck severity (FHWA/AASHTO). Field measurement protocols (ACI 224R, ASTM C876, ASTM C856).

**Purpose:** Primary seed dataset for fine-tuning a **crack classification copilot**. Each answer uses correct engineering terminology, cites specific standards, and gives actionable field recommendations. Small enough for initial QLoRA experiments; high enough quality to anchor the model's domain voice.

### second-brain-train — 75 records
General knowledge Q&A pairs for broad reasoning capability.

**Purpose:** Prevents catastrophic forgetting during fine-tuning. Mixed in with domain data so the model retains general instruction-following while acquiring construction expertise.

## Synthetic Datasets (Live Generation)

All synthetic data is generated by free Gemma 4 models via the Google AI API, deduplicated by SHA-256 content hash against 67K+ existing entries, and tagged with `_meta` (stream name, model, topic, timestamp, iteration number).

### construction_qa — ~4,650 records
**What it contains:** Multi-turn conversations between a field inspector and a senior engineer. Covers real job-site scenarios — concrete QC failures, structural assessments, repair decisions, code compliance questions. Topics are anchored to Illinois/Chicago projects (ISU dormitories, Loop high-rises, suburban infrastructure).

**Purpose:** High-volume chat SFT data. Teaches the model to reason through field problems, cite relevant codes (ACI, ASTM, IBC, Chicago Building Code), assess severity, and give prioritized recommendations. This is the largest chat-format training source for the crack/inspection assistant.

### blueprint_interpretation — ~9,800 records
**What it contains:** Structured entries with `drawing_type`, `sheet_reference`, `conversation` (user asks about a drawing detail, assistant explains), `symbols_referenced`, `codes_referenced`, and `difficulty_level`. Covers structural, mechanical, electrical, and architectural drawings.

**Purpose:** Teaches the model to **interpret engineering drawings from text descriptions**. No public dataset exists for this. Enables queries like "What does the dashed line on S-201 mean?" and getting back the correct structural explanation with code references. Foundation for a future multimodal system where users photograph a drawing and get explanations.

### safety_inspections — ~6,840 records
**What it contains:** Complete OSHA-style inspection reports with `site_name`, `location`, `findings[]` (each with OSHA standard number, observation, severity, corrective action, abatement deadline), `compliant_items`, `overall_rating`, and `inspector_notes`.

**Purpose:** Fine-tunes the model to **generate structured inspection reports from field observations**. Each finding maps to a specific OSHA regulation (e.g., 1926.1424 for crane operations, 1926.251 for rigging). Directly applicable to automated documentation tools for safety officers. Aligns with OSHA Susan Harwood grant requirements.

### building_codes — ~5,220 records
**What it contains:** Code compliance Q&A structured with `jurisdiction`, `code_reference`, `conversation`, `applicable_codes[]` (code name, section, requirement text), `practical_implications`, and `common_mistakes`. Covers IBC, IEBC, Chicago Building Code (CBC Title 14B), and Illinois amendments.

**Purpose:** Teaches the model to **navigate overlapping building code jurisdictions** — which code applies, which section controls, and what contractors commonly get wrong. Specialized for the Chicago/Illinois regulatory environment. Useful for both a code lookup assistant and automated plan review support.

### materials_database — ~8,900 records
**What it contains:** Material specification cards with `properties` (strength, density, fire rating, thermal conductivity), `astm_standards`, `testing_methods[]` (test name, standard, frequency, acceptance criteria), `typical_applications`, `illinois_considerations`, `common_defects`, and `sustainability_data`.

**Purpose:** Creates a **queryable materials knowledge base**. A fine-tuned model can answer "What test do I run to verify this concrete meets spec?" with the exact ASTM number, frequency, and pass/fail criteria. The `illinois_considerations` field adds region-specific warnings (freeze-thaw, deicing salt exposure) that generic materials databases miss.

### disaster_assessment — ~5,450 records
**What it contains:** Post-disaster structural evaluations with `event_type` (earthquake, flood, fire, tornado, explosion), `structural_evaluation` (foundation, superstructure, lateral system, roof, nonstructural), `damage_classification`, `placard` (green/yellow/red per ATC-20), `safety_concerns`, and `recommended_actions` (prioritized).

**Purpose:** Trains a **rapid post-disaster triage assistant**. The model learns the FEMA/ATC-20 placard system, how to prioritize immediate safety actions vs. engineering follow-ups, and how to structure findings for emergency management. Directly relevant to DOT emergency bridge assessment protocols and FEMA BRIC grant applications.

### image_descriptions — ~10,730 records
**What it contains:** Rich scene descriptions with `objects[]` (label, region, condition, material, notes), `anomalies[]` (type, severity, location, description), `safety_observations`, `caption_short`, and `caption_detailed`. Each entry describes a construction scene as if annotating a photograph.

**Purpose:** **Vision-language alignment data**. Even without paired images, these detailed text descriptions teach the model what to look for in construction photos — PPE compliance, structural anomalies, equipment condition. Can be used as:
1. Caption training data when paired with real or generated images
2. Text-only SFT for "describe what you see" construction scene analysis
3. Anomaly detection training (each entry flags what's wrong and how severe)

### object_detection_labels — ~14,730 records
**What it contains:** COCO-style annotations with `scene_id`, `scene_type`, `annotations[]` (category, bounding box description, relative position, attributes including `condition` and `is_damaged`), organized by supercategories (person, PPE, equipment, structural, hazard).

**Purpose:** **Object detection and safety compliance training**. The `is_damaged` and `condition` attributes go beyond simple object presence — they capture whether PPE is worn correctly, whether equipment is in safe condition, and whether hazards are present. This data can train or evaluate:
1. PPE compliance detectors (hard hat, harness, hi-vis, gloves)
2. Equipment condition classifiers (corroded, damaged, missing tags)
3. Hazard identification models (unbarricaded zones, tripping hazards)

## Benchmark Prompt Sets

| File | Scenarios | Purpose |
|------|-----------|---------|
| smoke.jsonl | 2 | Basic validity — does the model respond coherently and produce valid JSON? |
| qa.jsonl | 2 | General Q&A comprehension baseline |
| safety-benchmark.jsonl | 10 | Safety constraint adherence — refusal, guardrails, keyword compliance |
| safety-reasoning.jsonl | 5 | Reasoning under safety constraints |
| stress-standard.jsonl | 3 | Sustained load testing at high throughput |
| stress-reasoning.jsonl | 3 | Heavy reasoning under load |

**Purpose:** Standardized eval harness for comparing models before and after fine-tuning. The safety prompts are especially important — they produced the finding that Gemma 4 E2B (2.3B effective params) outscores E4B (4.5B) on safety compliance.

## How It All Fits Together

```
                    ┌─────────────────────────┐
                    │  Fine-Tuning Pipeline    │
                    └────────┬────────────────┘
                             │
        ┌────────────────────┼────────────────────┐
        │                    │                    │
  ┌─────▼─────┐      ┌──────▼──────┐     ┌──────▼──────┐
  │  Chat SFT  │      │ Structured  │     │   Vision    │
  │            │      │ Output SFT  │     │  Alignment  │
  └─────┬─────┘      └──────┬──────┘     └──────┬──────┘
        │                    │                    │
   cracks (75)          safety_insp (6.8K)   image_desc (10.7K)
   construction_qa      building_codes (5.2K) obj_detect (14.7K)
     (4.6K)             materials_db (8.9K)
   second_brain (75)    disaster (5.4K)
                        blueprint (9.8K)
        │                    │                    │
        └────────────────────┼────────────────────┘
                             │
                    ┌────────▼────────────────┐
                    │  Evaluation (25 prompts) │
                    │  smoke / qa / safety /   │
                    │  stress / reasoning      │
                    └─────────────────────────┘
```

**Chat SFT** teaches conversational domain expertise. **Structured output SFT** teaches the model to produce machine-parseable reports, material cards, and code citations. **Vision alignment** prepares for multimodal deployment where field photos feed into the system. The benchmark prompts measure whether fine-tuning improved domain performance without degrading safety or general capability.
